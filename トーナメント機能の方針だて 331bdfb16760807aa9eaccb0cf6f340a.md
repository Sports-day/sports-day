# トーナメント機能 実装設計方針書

## 本ドキュメントの目的

トーナメント機能の新規実装にあたり、DB設計・ロジック設計の方針を実装者に共有するためのドキュメント。
既存サービス（Challonge、Start.gg、Toornament、brackets-manager.js）の設計を調査・比較した上で、
**既存のDBスキーマ・リーグ機能・自動進出機能との整合**を取った設計方針をまとめている。

技術スタック: MySQL, Go (GORM + リポジトリ層), Go GraphQL

**実装方針:**

- **リポジトリ層:** 基本的なCRUD操作と、必要に応じて特殊なバッチ処理を実装する。DB操作は既存のリポジトリ層の関数を利用し、不足があれば追加する
- **Service層:** ビジネスロジック（自動進行、バリデーション、進出処理等）と**トランザクション管理**を担当する。複数のリポジトリ層関数をトランザクション内でまとめて呼び出す形で実装する
- 以下のモデル定義はデータ構造と制約を示すもので、具体的なORMタグや実装方法は実装者が既存コードに合わせて判断すること

## 1. 既存スキーマの分析

### 1-1. 既存テーブル構造の要約

```
users ─────────────────── ユーザー（VARCHAR(26) PK = ULID）
groups ────────────────── クラス等のグループ
  └── group_users ─────── グループ ↔ ユーザー（多対多）
teams ─────────────────── チーム（→ groups に所属）
  └── team_users ──────── チーム ↔ ユーザー（多対多）
sports ────────────────── 競技マスタ
locations ─────────────── 会場マスタ

competitions ──────────── 大会（type: 'LEAGUE' | 'TOURNAMENT'）
  ├── competition_entries ── 大会 ↔ チーム（エントリー）
  ├── matches ─────────── 試合（→ competition, → location, → winner_team）
  │     ├── match_entries ── 試合 ↔ チーム（スコア付き、team_id nullable）
  │     └── judgments ────── 審判情報
  └── leagues ─────────── リーグ拡張（competitions.id と共有PK、1:1）
        └── league_standings ── 順位表（※要件書により削除予定）
```

### 1-2. 設計上の重要な既存パターン

**パターン1: leagues の共有PKパターン**

`leagues` テーブルは `competitions.id` をPKかつFKとして持ち、competitionを1:1で拡張している。
**トーナメントはこのパターンを採用しない。** 1つのcompetitionに複数のブラケットツリーを持てるようにするため、
`tournaments` は `competition_id` をFKとして持つ**1:N関係**にする。

この多重度の違いはドメインの性質を反映している。リーグの「予選A」「予選B」は別のcompetitionだが、
トーナメントの「メインブラケット＋3位決定戦＋5位決定戦」は1つの大会の中にある構造のため。

**パターン2: matches は competition に所属**

`matches.competition_id` により、リーグの試合もトーナメントの試合も同じテーブルに入る。

**パターン3: match_entries がスロット的役割**

各試合に対して参加チーム分の `match_entries` が存在し、`team_id` がnullable（未確定対応済み）。

**パターン4: IDはVARCHAR(26) = ULID**

全テーブルで統一されている。新規テーブルもこれに従う。

## 2. 設計思想

### 2-1. スロットグラフモデル

トーナメントを**試合（Match）とスロット（Slot）のDAG（有向非巡回グラフ）**として表現する。

- 各試合は**2つのスロット**を持つ（対戦する2チーム分）
- 各スロットには**「何がここに入るか」のソース情報**がある
- ソースは3種類: **SEED（シード配置）/ MATCH_WINNER（前試合の勝者）/ MATCH_LOSER（前試合の敗者）**

### 2-2. 既存サービスからの採用要素

| 採用元 | 採用した要素 | 理由 |
| --- | --- | --- |
| Start.gg | SetSlotの `prereqType` + `prereqId` + `prereqPlacement` | スロットのソースを統一的に表現できる |
| Toornament | `source_type` + `source_node_id` | 勝者/敗者をソースとして指定する考え方 |
| brackets-manager.js | 1つのStage内に複数のGroup（Winner/Loser/Final）を持つ構造 | マルチブラケット設計の参考 |
| Challonge | `is_prereq_match_loser` パターン | `MATCH_LOSER` に相当 |

### 2-3. マルチブラケットアーキテクチャ

1つのトーナメント大会（`competitions`）の中に、**複数の独立したブラケットツリー（`tournaments`）**を持てるようにする。

```
competitions (type='TOURNAMENT')  ← UIで見える「1つのトーナメント」
  ├── tournaments (メインブラケット)        bracket_type=MAIN, display_order=1
  ├── tournaments (3位決定戦ブラケット)     bracket_type=SUB, display_order=2
  └── tournaments (5-8位決定戦ブラケット)   bracket_type=SUB, display_order=3
```

- **UIレベル**: ユーザーからは1つのトーナメントに見える
- **データレベル**: 独立したブラケットツリーの集合
- **ツリー間接続**: `tournament_slots.source_match_id` が同一competition内の別ブラケットの試合も参照できる

**この設計のメリット:**

- `has_third_place_match` のようなブーリアンフラグが不要。あらゆるサブブラケットを汎用的に作れる
- 各ブラケットが独立した状態を持てる（メインが終わってからサブを開始、等）。状態はDBカラムではなく試合のステータスから導出する
- ダブルエリミネーション構造にも対応可能（MAIN + SUBブラケット複数）

**既存サービスでの先例:**

- **brackets-manager.js**: ダブルエリミネーションを「1つのStage内に3つのGroup」として表現
- **Start.gg**: 1つのPhase内に複数のPhaseGroupを持つ構造

## 3. テーブル設計

### 3-1. 方針

| テーブル | 方針 |
| --- | --- |
| `competitions` | **そのまま利用**。`type = 'TOURNAMENT'` がUIレベルのトーナメント |
| `competition_entries` | **そのまま利用**。トーナメントへのチームエントリー |
| `matches` | **そのまま利用**。変更なし |
| `match_entries` | **そのまま利用** |
| `tournaments` | **新規作成**。個別のブラケットツリー。competitionsと1:N |
| `tournament_slots` | **新規作成**。スロットのソース情報 |

### 3-2. tournaments テーブル（新規）

```go
type Tournament struct {
	ID              string     // VARCHAR(26), PK
	CompetitionID   string     // VARCHAR(26), NOT NULL, FK → competitions(id) ON DELETE CASCADE
	Name            string     // VARCHAR(64), NOT NULL（ブラケット名。例: メイン、3位決定戦）
	BracketType     string     // VARCHAR(16), NOT NULL, DEFAULT 'MAIN'（MAIN / SUB）
	PlacementMethod *string    // VARCHAR(16), NULL（MAINブラケットで設定。SUBではNULL）
	DisplayOrder    int        // INT, NOT NULL, DEFAULT 1（UI上の表示順）
	CreatedAt       time.Time
	UpdatedAt       time.Time
}
```

**インデックス:**

- `CompetitionID` にインデックス

**制約（Service層のバリデーションで保証）:**

- `BracketType` は `"MAIN"` または `"SUB"` のいずれか
- `PlacementMethod` は `nil`（SUBブラケット）、または `"SEED_OPTIMIZED"` / `"BALANCED"` / `"RANDOM"` / `"MANUAL"` のいずれか
- 同一 `CompetitionID` 内に `BracketType = "MAIN"` は1つだけ

**bracket_type の値:**

| 値 | 用途 |
| --- | --- |
| `MAIN` | メインブラケット。外部からの進出（SEEDスロット）を受け入れる |
| `SUB` | サブブラケット。3位決定戦、5-8位決定戦、敗者復活戦、ダブルエリミの敗者ブラケット等すべて |

**placement_method の値:**

| 値 | 用途 |
| --- | --- |
| `SEED_OPTIMIZED` | 上位シード同士がブラケット終盤まで当たらないように配置 |
| `BALANCED` | 上位と下位が序盤で対戦する対称配置 |
| `RANDOM` | シードをランダムに割り当て |
| `MANUAL` | 各 `promotion_rules` レコードの `slot` で個別にスロットを指定 |
| `NULL` | SUBブラケット（SEEDスロットがないため配置不要）、またはリーグ進出先 |

3位決定戦とダブルエリミの敗者ブラケットをDB上で区別する必要はない。どちらも「MAINから MATCH_LOSER でチームを受け取るサブブラケット」であり、振る舞いの違いは slot の接続パターンで表現される。

**⚠ MAINブラケットは1つのcompetitionに対して1つだけ。** アプリケーション層で、同一competition内に `bracket_type = 'MAIN'` のtournamentsが複数作成されないようバリデーションすること。SUBブラケットは複数作成可能。

**トーナメントのリセット:**

トーナメント全体をやり直す場合は、セクション3-4のブラケット削除手順に従い、全 tournaments を先に削除してから matches を削除する。その後、ブラケットを再構築する。特別なリセット用APIやステータスは設けない。

**設計判断: status カラムを持たない**

既存の `leagues` テーブルにもステータスカラムはない。リーグの状態は試合のステータス（STANDBY/ONGOING/FINISHED）から導出している。トーナメントも同じパターンに統一する。

ブラケットの状態は以下で導出する:

| 状態 | 導出条件 |
| --- | --- |
| 構築中 | `source_type = 'SEED'` のスロットに紐づく match_entry のうち `team_id = NULL` のものがある |
| 準備完了 | 全SEEDスロットの match_entry に team_id がある、かつ全試合が未開始 |
| 進行中 | 所属 matches のうち1つ以上が ONGOING or FINISHED |
| 完了 | 所属 matches が全て FINISHED |

**注意:** MATCH_WINNER / MATCH_LOSER スロットの match_entry は、前の試合が完了するまで team_id = NULL が正常。「構築中」の判定は SEED スロットのみを対象とする。

ブラケット編集の制限（構築中のみ編集可能）は、「所属 matches にスコアが1つでも入っていれば編集不可」で判定する。明示的な status カラムによる制御は不要。

### 3-3. tournament_slots テーブル（新規）★設計の核心

各 `match_entry` に対するトーナメント固有のソース情報を管理する。
`leagues` が `competitions` の拡張テーブルであるのと同様、`tournament_slots` は `match_entries` の拡張テーブルとして機能する。

```go
type TournamentSlot struct {
	ID            string     // VARCHAR(26), PK
	TournamentID  string     // VARCHAR(26), NOT NULL, FK → tournaments(id) ON DELETE CASCADE
	MatchEntryID  string     // VARCHAR(26), NOT NULL, FK → match_entries(id) ON DELETE CASCADE, UNIQUE
	SourceType    string     // VARCHAR(16), NOT NULL（SEED / MATCH_WINNER / MATCH_LOSER）
	SourceMatchID *string    // VARCHAR(26), NULL, FK → matches(id) ON DELETE RESTRICT
	SeedNumber    *int       // INT, NULL（SEED時のみ設定）
	CreatedAt     time.Time
	UpdatedAt     time.Time
}
```

**インデックス:**

- `TournamentID` にインデックス
- `SourceMatchID` にインデックス

**ユニーク制約:**

- `MatchEntryID`: 1つの match_entry に対して tournament_slot は1つだけ（1:1保証）
- `(TournamentID, SeedNumber)`: 同じブラケット内でシード番号は重複不可。NULLは除外される

**FK削除戦略:**

- `TournamentID` → CASCADE（ブラケット削除でスロットも消える）
- `MatchEntryID` → CASCADE（match_entry 削除でスロットも消える）
- `SourceMatchID` → RESTRICT（参照先の試合は削除できない。先にスロットの接続を解除する）

**制約（Service層のバリデーションで保証）:**

- `SourceType` は `"SEED"` / `"MATCH_WINNER"` / `"MATCH_LOSER"` のいずれか
- `SourceMatchID` が同一competition内の試合を参照していること（セクション3-3の必須バリデーション参照）

**設計判断: なぜ match_entries に直接カラムを追加せず別テーブルにするか**

`match_entries` はリーグでも使われている汎用テーブル。トーナメント固有のカラム（source_type, source_match_id, seed_number）をリーグの試合にも持たせるのは不自然。`leagues` が `competitions` を拡張するのと同じパターンで、`tournament_slots` が `match_entries` を1:1拡張する形にする。

**設計判断: なぜ `match_id` カラムを持たないか**

試合IDは `match_entry_id` → `match_entries.match_id` で辿れるため冗長。JOINのコストは `match_entry_id` がユニークキーなので低い。冗長カラムによるデータ不整合リスクを排除する。

**★ `source_match_id` のスコープと必須バリデーション:**

`source_match_id` は**同一competition内の任意の試合**を参照できる。同一ブラケット内に限定されない。
これにより、メインブラケットの準決勝の敗者を3位決定戦ブラケットに送る、といった**ツリー間接続**が実現できる。

FKは `matches(id)` に対して設定されており、ブラケットをまたいだ参照もDB制約上問題ない。

**⚠ ただし、「`source_match_id` が同一competition内の試合であること」はDB制約では保証できない。**
別competitionの試合を参照するレコードがFK上は作成可能なため、**アプリケーション層で以下のバリデーションを必ず実装すること:**

```
source_match_id が指す match の competition_id
  == このスロットが属する tournament の competition_id
```

このバリデーションはスロット作成時・接続変更時の両方で実行する。

**設計判断: `source_match_id` の ON DELETE RESTRICT**

`source_match_id` のFKは `ON DELETE RESTRICT` とする（SET NULL や CASCADE ではない）。
他のスロットから参照されている試合は削除できない。削除したい場合は、先に参照元スロットの接続を解除（source_match_id を別の試合に変更、またはスロット自体を削除）してから試合を削除する。
SET NULL にすると「ソースが不明なスロット」という壊れた状態が発生するため、DB制約レベルで防止する。

**source_type の3値:**

| source_type | 意味 | 使用場面 |
| --- | --- | --- |
| `SEED` | 外部から配置されるチーム。このスロットの `match_entry.team_id` に直接書き込まれる | 初戦のスロット。進出システムから配置される |
| `MATCH_WINNER` | `source_match_id` の試合の勝者が、このスロットの `match_entry.team_id` に書き込まれる | 通常は同一ブラケット内の進行。ダブルエリミネーション等ではブラケット間でも使用可能 |
| `MATCH_LOSER` | `source_match_id` の試合の敗者が、このスロットの `match_entry.team_id` に書き込まれる | **ブラケット間接続**の主な手段。3位決定戦、敗者復活戦等 |

**ユニーク制約:**

- `uidx_ts_match_entry`: 1つの match_entry に対して tournament_slot は1つだけ（1:1保証）
- `uidx_ts_seed`: 同じブラケット内でシード番号は重複不可（NULLは除外される）

**⚠ トーナメント試合の作成パターン（必須）:**

トーナメントの試合を作成する際は、以下の5レコードを**必ずセットで作成**すること。個別に作成すると整合性が壊れる。

```
1. Match レコード（1つ）
2. MatchEntry レコード（2つ）
3. TournamentSlot レコード（2つ: 上記 MatchEntry それぞれに対応）
```

Service層のメソッドとしてこの5レコードのセット作成を実装し、トーナメント試合の唯一の作成手段とすること。Service層でトランザクションを開始し、リポジトリ層の関数を呼び出して一括作成する。

### 3-4. 既存テーブルへの変更

**既存テーブルへの変更はなし。** `matches` / `match_entries` / `competitions` はそのまま利用する。

`placement_method` は `tournaments` テーブル（セクション3-2）に持たせる。`competitions` テーブルには追加しない。

トーナメントの試合も既存の `matches` テーブルに `competition_id` で紐づけて格納する。
ブラケットとの紐付きは `tournament_slots.match_entry_id` → `match_entries.match_id` で辿る。

**あるブラケットの全試合を取得するクエリ:**

```sql
SELECT DISTINCT m.* FROM matches m
JOIN match_entries me ON me.match_id = m.id
JOIN tournament_slots ts ON ts.match_entry_id = me.id
WHERE ts.tournament_id = ?
```

学内球技大会の規模（ブラケットあたり最大十数試合）ではこのJOINでパフォーマンス問題は起きない。

**敗者の導出方法:**

`loser_team_id` カラムは持たない。敗者は既存の情報から導出する:

```
敗者 = match_entries のうち team_id ≠ matches.winner_team_id である側の team_id
```

2チームの試合では `winner_team_id` が確定すれば敗者は一意に決まるため、非正規化は不要。

**ブラケット削除時の試合削除:**

`matches` に `tournament_id` を持たないため、CASCADEによる連鎖削除はできない。
ブラケット削除時はアプリケーション層で以下の順序で削除する:

```
1. 対象ブラケットの tournament_slots を取得し、紐づく match_entries.match_id を収集
2. tournaments レコードを削除（tournament_slots が CASCADE で連鎖削除される）
   → これにより source_match_id の RESTRICT 参照が解除される
3. 収集した match_ids の matches を削除（match_entries は CASCADE で連鎖削除される）
```

**⚠ ステップ2と3の順序が重要。** matches を先に削除しようとすると、tournament_slots.source_match_id の ON DELETE RESTRICT により削除がブロックされる。先に tournaments を削除して tournament_slots を CASCADE で消すことで、RESTRICT の参照が解除され、matches の削除が可能になる。

**⚠ 複数ブラケットがある場合:** SUBブラケットの tournament_slots が MAINブラケットの matches を source_match_id で参照している場合がある。全ブラケットを削除する（トーナメントリセット）場合は、**全 tournaments を先に削除**してから matches を削除すること。

## 4. テーブル間の関係図

```
competitions (type='TOURNAMENT')
  │
  ├──< competition_entries (エントリー)
  │       └── team_id → teams
  │
  ├──< tournaments (1:N、ブラケットツリーの集合)
  │       bracket_type (MAIN / SUB)
  │       placement_method (MAINで設定、SUBはNULL)
  │       display_order
  │       │
  │       └──< tournament_slots (★ソース情報、ブラケットに直接紐づく)
  │              match_entry_id → match_entries (1:1)
  │              source_type, source_match_id, seed_number
  │              ※source_match_id は別ブラケットの試合も参照可
  │
  └──< matches (試合、competition_id で大会に紐づく)
          │  winner_team_id
          │
          ├──< match_entries (チーム・スコア)
          │      team_id (nullable), score
          │
          └──< judgments (審判情報)
```

**ブラケット間接続の例（4チーム + 3位決定戦）:**

```
competition: バスケ決勝トーナメント
│
├── tournament A (MAIN, "メインブラケット")
│     match_sf1: seed 1 vs seed 4
│     match_sf2: seed 2 vs seed 3
│     match_final: match_sf1の勝者 vs match_sf2の勝者
│
└── tournament B (SUB, "3位決定戦")
      match_3rd: match_sf1の敗者 vs match_sf2の敗者
                 ↑ source_match_id が tournament A の試合を参照
```

# トーナメント機能 実装設計方針書（2/3: ロジック編）

## 5. 既存の進出システムとの接続

### 5-1. promotion_rules テーブルとの対応

リーグ要件書で定義された `promotion_rules` テーブルは以下の構造を持つ:

| カラム | 内容 |
| --- | --- |
| source_competition_id | 進出元の大会ID |
| target_competition_id | 進出先の大会ID |
| rank_spec | 進出元の何位を送るかの指定。 `"1"` = 1位のみ / `"1,3"` = 1位と3位 / `"1-4"` = 1位〜4位。進出先が手動配置の場合は単体指定のみ |
| slot | 進出先の seed_number に対応。進出先の配置方法が MANUAL の場合のみ使用。それ以外はNULL |

配置方法（`placement_method`）は `promotion_rules` ではなく **`tournaments`（ブラケット）テーブル**に持つ。
MAINブラケットで設定し、SUBブラケットではNULL（SEEDスロットがないため配置不要）。

トーナメントが進出先の場合、各カラムは以下のように `tournament_slots.seed_number` と対応する:

**MANUAL 配置の場合:**

`promotion_rules.slot` の値が `tournament_slots.seed_number` に直接対応する。
手動配置では `rank_spec` は単体指定のみ（1レコード = 1順位 × 1スロット）。
例: `rank_spec = "1", slot = 3` なら、1位チームが seed_number=3 に配置される。

**プリセット配置（SEED_OPTIMIZED / BALANCED / RANDOM）の場合:**

プリセットは、進出チームを seed_number にどう割り当てるかを決めるアルゴリズム。
**全チームが揃った時点（期待チーム数到達時）で**スロット割当を実行する。
割り当て後は `tournament_slots.seed_number` に対応する match_entry に team_id を書き込む。

| プリセット | seed_number の割り当て方法 |
| --- | --- |
| SEED_OPTIMIZED | 上位シード同士がブラケット終盤まで当たらないように配置。1位と2位は反対ブロック |
| BALANCED | シード順で対称的に配置。上位と下位が序盤で対戦する形 |
| RANDOM | seed_number をランダムに割り当て |

複数リーグから進出する場合のシード順: エントリーの作成順（ULID順）でシード番号を1から振る。

**リーグ進出の場合:**

進出先がリーグ（`competitions.type = 'LEAGUE'`）の場合、`tournaments` テーブルが存在しないため `placement_method` は関係ない。スロット配置は不要で、`competition_entries` にチームを追加するだけ。

### 5-2. ブラケットの特定方法

進出先がトーナメント（`competitions.type = 'TOURNAMENT'`）の場合、SEEDスロットの検索は以下で行う:

```sql
SELECT ts.* FROM tournament_slots ts
JOIN tournaments t ON ts.tournament_id = t.id
WHERE t.competition_id = ?          -- promotion_rules.target_competition_id
  AND t.bracket_type = 'MAIN'
  AND ts.source_type = 'SEED'
  AND ts.seed_number = ?            -- 割り当てられた seed_number
```

**seed_number はブラケット（tournaments）単位でユニーク。**
通常は MAINブラケット のSEEDスロットに対して進出ルールを設定する。
SUBブラケットにはブラケット間接続（MATCH_LOSER）でチームが入るため、外部からの進出ルールは通常不要。

### 5-3. 進出ステップ1（エントリー作成）での処理

リーグ要件書セクション4-4「ステップ1: エントリー作成」と同じ流れ。

1. 進出ルールに従い、対象順位のチームを特定する
2. `competition_entries` にチームを追加する
3. 進出先の `placement_method` が MANUAL の場合:
    - `promotion_rules.slot` で指定された seed_number に対応する `tournament_slots` を検索
    - 紐づく `match_entries.team_id` にチームIDを書き込む
4. 進出先の `placement_method` がプリセットの場合:
    - この時点ではスロット割当は行わない（全チームが揃ってから実行する）

### 5-4. 進出ステップ2（トーナメント固有の動作）

リーグ要件書セクション4-4「ステップ2: 試合生成」に対応するが、**トーナメントでは試合は「生成」ではなく「開始可能化」**。

エントリー数が期待チーム数に達した時点で:

1. 進出先の `placement_method` がプリセットの場合、全チームに対してプリセットのロジックでスロット（seed_number）を割り当て、`match_entries.team_id` を書き込む
2. トーナメントの場合: ブラケット構造は**事前に存在**している。全SEEDスロットが埋まった時点で、初戦が**開始可能**になる
3. リーグの場合: 対戦カード（試合）を**自動生成**する

SUBブラケットはMAINブラケットの試合結果に依存するため、MATCH_LOSER でチームが配置されるまで試合は開始できない。

### 5-5. 期待チーム数の算出

```
期待チーム数 = MAINブラケットの tournament_slots で source_type = 'SEED' のレコード数
```

リーグ要件書では「期待チーム数 = promotion_rules の rank_spec から算出される自動エントリー数 + 手動エントリー数」と定義されている。

**⚠ この2つの数値は一致しなければならない。**

不一致があるとブラケットが永久に開始できない（SEEDスロットが余る）か、エントリーされたチームの配置先がない（SEEDスロットが足りない）。

バリデーション:

```
promotion_rules の rank_spec から算出される自動エントリー数
+ 手動エントリー数
== MAINブラケットのSEEDスロット数
```

このチェックは進出ルール設定時・ブラケット構造変更時の両方で実行する。

### 5-6. 手動エントリーの扱い

リーグ要件書セクション4-5に従い、**手動エントリーの追加・削除も進出先の状態によって制限する。**

トーナメントが進出先の場合、手動エントリーとは「管理者が進出ルールを介さずに直接SEEDスロットにチームを配置する操作」に相当する。

| 進出先の状態 | 手動エントリー |
| --- | --- |
| 全試合未開始 | 追加・削除可能 |
| 試合のスコアが1つでも入力されている | 追加・削除不可 |

### 5-7. 進出ルール変更時のトーナメント側の動作

リーグ要件書セクション4-5に従い、進出ルールの変更はトーナメント（進出先）の状態によって制限する。

| 進出先の状態 | ルール変更 |
| --- | --- |
| 全試合未開始 | **変更可能** |
| 試合のスコアが1つでも入力されている | **変更不可** |

変更可能な状態でルールが変更された場合:

1. 既存のSEEDスロットの `match_entries.team_id` をすべて NULL に戻す
2. 期待チーム数を再算出し、SEEDスロット数との整合を再チェック
3. 新しい進出ルールに従ってチームを再配置

**注意:** リーグが進出先の場合は「エントリーや試合を再構成」だが、トーナメントが進出先の場合は「スロットの team_id をクリアして再配置」。ブラケット構造（matches + tournament_slots）自体は変更しない。

### 5-8. トーナメントを進出元にする場合

**注: リーグ要件書セクション4-2ではトーナメント→トーナメント / トーナメント→リーグの進出パターンは「将来対応（拡張可能な設計にしておく）」とされている。以下は将来実装時の設計方針を定義するもの。**

トーナメントが進出元（子）の場合、進出ルールは **competition_id** に対して設定する。
個別のブラケット（tournaments）ではなく、competition 単位で順位が確定するため。

```
promotion_rules.source_competition_id = competition.id (type='TOURNAMENT')
  → competitionの順位（全ブラケットの結果から導出）の N 位を進出先に送る
```

これは既存のリーグの進出ルール（source_competition_id = league.id = competition.id）と同じ粒度。
`promotion_rules` テーブルのスキーマ変更は不要。

## 6. トーナメント内の自動進行

### 6-1. 基本フロー

試合のスコアが確定した（`matches.status = 'FINISHED'`）時点で、以下を順に実行する:

1. 勝者を `matches.winner_team_id` に記録（既存の仕組み）
2. 敗者を導出（`match_entries` のうち `team_id ≠ winner_team_id` の側）
3. この試合IDを `source_match_id` として参照している `tournament_slots` を検索
4. 各スロットの `source_type` に応じて:
    - `MATCH_WINNER` → 勝者を配置
    - `MATCH_LOSER` → 敗者を配置
5. チーム配置されたスロットの試合のステータスを更新
6. **competition内の全ブラケットの全試合が FINISHED かチェック**し、全完了であれば `promotion_rules` に基づくステージ間自動進出を開始する（セクション6-4参照）

ステップ6はリーグ要件書セクション3-4と同じ考え方: 「毎回のスコア入力時に全試合の状態をチェックし、全完了であれば進出処理を開始する」。トーナメントでもスコア入力ごとに毎回チェックする。

### 6-2. ブラケット間自動進行

**同一ブラケット内の進行と全く同じロジックで動作する。**

`source_match_id` による検索はブラケットをまたいで行われるため、
メインブラケットの準決勝1が完了すると、以下が同時に起きる:

- メインブラケットの決勝のスロットに勝者が配置される（`MATCH_WINNER`）
- **3位決定戦ブラケットのスロットに敗者が配置される**（`MATCH_LOSER`）

追加のロジックは不要。スロットグラフモデルがブラケット境界を透過的に扱える。

### 6-3. ブラケットの状態（導出）

ブラケットの状態は `tournaments` テーブルにカラムを持たず、所属する試合とスロットの状態から導出する（`leagues` にステータスカラムがないのと同じパターン）。導出条件はセクション3-2を参照。

**SUBブラケットの状態遷移:**

SUBブラケットにはSEEDスロットが通常存在しない（全スロットが MATCH_LOSER）。
MAINブラケットの対応する試合が完了し、`MATCH_LOSER` でチームが配置された時点で
SUBブラケットの試合が開始可能になる。SUBブラケットは「構築中 → 進行中」を直接遷移する（「準備完了」を経由しない場合がある）。

### 6-4. competition全体の完了判定

**competition内の全ブラケットが「完了」状態になった時点で**、competition全体が完了とみなされる。

完了判定のクエリ:

```sql
SELECT COUNT(*) FROM tournaments t
JOIN tournament_slots ts ON ts.tournament_id = t.id
JOIN match_entries me ON me.id = ts.match_entry_id
JOIN matches m ON m.id = me.match_id
WHERE t.competition_id = ?
  AND m.status != 'FINISHED'
```

結果が0なら全試合完了。

1. 全ブラケットの全試合が FINISHED であることを確認
2. competition全体の順位を確定する（セクション7参照）
3. **ステージ間の自動進出をトリガー**（トーナメント→トーナメント / トーナメント→リーグ）

ただし、進出ルールが参照する順位のチームが確定していれば、
一部のSUBブラケットが未完了でもトリガー可能とする。
具体的には、`promotion_rules.rank_spec` で指定された順位がMAINブラケットの結果だけで確定し、
未完了のSUBブラケットがその順位の解消に関与しない場合に限る。

例: `rank_spec = "1-2"` で、MAINブラケットの決勝が完了していれば、3位決定戦（SUBブラケット）が未完了でも進出トリガー可能。
例: `rank_spec = "1-3"` の場合、3位を確定させるSUBブラケットが完了するまでトリガー不可。

**設計判断: ブラケット・competition ともにステータスカラムを持たない**

既存の `leagues` テーブルにもステータスカラムはなく、リーグの状態は試合のステータスから導出している。
トーナメントも同じパターンに統一する。ステータスカラムを持つと試合のステータスとの二重管理になり同期コストが発生する。

### 6-5. 引き分けの禁止

トーナメントの試合では引き分けを許可しない。自動進行には必ず勝者が必要なため。

**バリデーション:**

- トーナメントの試合（その試合の match_entries に紐づく tournament_slots が存在する試合）で `status` を `FINISHED` にする際、`winner_team_id` が設定されていることを必須とする
- 両チームのスコアが同点の場合、`FINISHED` にできない（延長戦・PK戦等の結果を反映してからFINISHEDにする）
- 管理者がスコアとは別に `winner_team_id` を手動指定する手段を用意する（PK戦等でスコアが同点のまま勝者が決まるケースに対応）

### 6-6. 不戦勝（棄権）の扱い

チームが棄権した場合、スコアなしで試合を完了させ、自動進行をトリガーする必要がある。

**処理フロー:**

1. 管理者が試合に対して「不戦勝」を宣言し、勝者チームを指定する
2. `matches.status` を `FINISHED`、`matches.winner_team_id` を勝者チームに設定する
3. `match_entries` のスコアは 0-0 のまま（または運用で決めた定型スコアを設定）
4. 以降は通常の自動進行と同じ: `source_match_id` を検索して後続スロットに勝者/敗者を配置

**⚠ 重要:** 不戦勝でも `winner_team_id` は必ず設定されるため、自動進行のロジックに特別な分岐は不要。「`FINISHED` かつ `winner_team_id` が設定済み」という同じ条件で自動進行がトリガーされる。

### 6-7. 同時実行制御

複数の管理者が同時にスコアを入力した場合、自動進行処理が競合する可能性がある。

**問題のシナリオ:**

```
管理者A: SF1 のスコア入力 → 決勝 slot1 に勝者配置
管理者B: SF2 のスコア入力 → 決勝 slot2 に勝者配置
  ↑ ほぼ同時に実行されると、決勝の「両スロット充足 → 試合開始可能」の判定が競合
```

**対策: 楽観的ロックを使用する**

`match_entries` の `team_id` 更新時に、現在の値が NULL であることを条件に含める（楽観的ロック）。更新件数が 0 なら、別のトランザクションが先に書き込んでいるため、リトライまたはエラーとする。この条件付き更新はリポジトリ層に実装する。

**⚠ 自動進行処理全体をService層のトランザクションで囲む:**

1つのスコア入力に対する自動進行処理（winner記録 → 後続スロット配置 → ブラケット間配置）は、Service層で1つのトランザクションを開始し、リポジトリ層の関数を呼び出す形で実行する。途中で失敗した場合は全体をロールバックする。

## 7. トーナメントの順位確定

### 7-1. 基本ルール

順位は**competition単位で、全ブラケットの結果を統合**して機械的に導出する。

**MAINブラケットからの順位:**

各ブラケットの「決勝」は、グラフ構造から導出する。具体的には**そのブラケット内で、自分を `source_match_id` として参照する `MATCH_WINNER` スロットが存在しない試合**がそのブラケットの最終試合（決勝）である。

| 条件 | 順位 |
| --- | --- |
| MAINブラケットの決勝の勝者 | 1位 |
| MAINブラケットの決勝の敗者 | 2位 |
| SUBブラケットなしの場合、MAINの準決勝敗者 | 同率3位 |
| 以降、各ラウンドの敗者 | 敗退ラウンドに応じた同率順位 |

**SUBブラケットがある場合:**

SUBブラケットの結果で同率順位が解消される。

- 3位決定戦ブラケットの決勝勝者 → 3位、敗者 → 4位
- 5-8位決定戦ブラケットの決勝勝者 → 5位、敗者 → 6位、準決勝敗者 → 同率7位

### 7-2. 順位導出のアルゴリズム

```
1. MAINブラケットの最終試合（MATCH_WINNERで後続されない試合）の勝者 → 1位、敗者 → 2位
2. 各SUBブラケットについて:
   a. そのブラケットの最終試合の勝者/敗者に順位を割り当てる
   b. 割り当てる順位は、そのブラケットに敗者を送った試合のラウンドから算出
      例: 準決勝の敗者が集まるブラケットなら 3位/4位
      例: 準々決勝の敗者が集まるブラケットなら 5位～8位
3. SUBブラケットでカバーされない敗者は同率順位のまま
```

### 7-3. 同率順位と進出ルールの関係

- SUBブラケットがない場合の同率順位は許容する
- **⚠ ただし、`promotion_rules` で参照される順位は一意に確定しなければならない。** 同率になりうる順位を `rank_spec` で指定することはバリデーションエラーとする

例: 4チームブラケットでSUBブラケット（3位決定戦）がない場合、3位は同率。この状態で `rank_spec = "3"` の進出ルールを設定しようとするとエラー。解決方法は「3位決定戦のSUBブラケットを作成する」。

バリデーション: 進出ルール設定時に、対象トーナメントのブラケット構造を検査し、`rank_spec` で指定された各順位が一意に確定する構造かどうかを確認する。

**リーグとの違い:**

リーグでは管理者裁定で同率が必ず解消される。トーナメントではブラケット構造（SUBブラケット）で解消する。管理者裁定の仕組みはトーナメントには適用しない。

## 8. ブラケットの自動生成 + 手動カスタマイズ

### 8-1. 自動生成

チーム数を入力すると、以下を自動生成する:

- `tournaments` レコード（MAINブラケット。オプションでSUB）
- `matches` レコード（各試合）
- `match_entries` レコード（各試合に2つ、team_id = NULL）
- `tournament_slots` レコード（各match_entryに対応、source情報付き）

### 生成オプション

| オプション | 説明 |
| --- | --- |
| チーム数 | 必須 |
| 3位決定戦 | ONにすると、SUBブラケット（準決勝敗者の1試合）を自動生成 |
| 5-8位決定戦 | ONにすると、SUBブラケット（準々決勝敗者の4チームトーナメント）を自動生成 |

これらはあくまで自動生成の利便性オプション。管理者は手動カスタマイズで任意のSUBブラケットを追加できる。

### 生成ロジック（MAINブラケット）

1. `rounds = ceil(log2(teamCount))` でラウンド数を算出
2. 決勝から逆算して試合を作成
3. 1回戦のスロットに `source_type = 'SEED'` + `seed_number` を割当
4. 2回戦以降のスロットに `source_type = 'MATCH_WINNER'` + `source_match_id` を設定

シード配置の標準パターン（8チーム）:

```
1回戦: seed 1 vs seed 8
1回戦: seed 4 vs seed 5
1回戦: seed 2 vs seed 7
1回戦: seed 3 vs seed 6
```

### 生成ロジック（SUBブラケット）

1. 別の `tournaments` レコードを `bracket_type = 'SUB'` で作成
2. 必要な試合を作成（3位決定戦なら1試合、5-8位決定戦なら3試合）
3. スロットに `source_type = 'MATCH_LOSER'` + MAINブラケットの対応試合の `match_id` を設定
4. **MAINブラケットの試合をまたいで `source_match_id` で参照する**

### BYE処理（2のべき乗でないチーム数の場合）

**BYE試合は作成しない。** 上位シードを次ラウンドのスロットに直接 `source_type = 'SEED'` で配置する。

BYEの数 = `次の2のべき乗 - チーム数`。BYEは上位シードに割り当てる。

**例: 5チームの場合**

BYEの数 = 8 - 5 = 3。seed 1, 2, 3 がBYE（1回戦免除）。

```
2回戦 slot(SEED, seed_number=1) ──────────┐
                                            ├── match(準決勝1)
1回戦 match: seed 4 vs seed 5             │
  slot(SEED, seed_number=4) ─┐             │
                               ├── match ──┘
  slot(SEED, seed_number=5) ─┘

2回戦 slot(SEED, seed_number=2) ──────────┐
                                            ├── match(準決勝2)
2回戦 slot(SEED, seed_number=3) ──────────┘
```

- 1回戦の試合は seed 4 vs seed 5 の1試合のみ
- seed 1, 2, 3 は2回戦のスロットに直接配置
- BYE専用ロジック（自動完了、除外判定等）が不要

**BYE配置のアルゴリズム:**

```
1. bracketSize = 次の2のべき乗に切り上げ（teamCount）
2. byeCount = bracketSize - teamCount
3. 上位byeCount個のシードを1回戦から除外
4. 除外されたシードは次ラウンドのスロットに source_type='SEED' で直接配置
5. 1回戦はBYEでないシード同士の対戦のみ作成
```

**期待チーム数との整合:**

```
期待チーム数 = MAINブラケット内の source_type='SEED' のスロット数
```

5チームの場合: seed 1〜5 のSEEDスロット = 5。正しい。

### 8-2. 手動カスタマイズ

ブラケットが「構築中」状態（所属試合にスコアが入っていない）の間、以下の操作が可能:

- **ブラケットの追加**: 新しいSUBブラケットを作成
- **ブラケットの削除**: 既存のSUBブラケットを削除（MAINは削除不可）
- **試合の追加・削除**: ブラケット内の試合を追加・削除
- **スロットの接続変更**: source_type, source_match_idの変更（ブラケット間接続を含む）
- **seed_numberの振り直し**

### 8-3. カスタマイズの制限

| ブラケットの状態（導出） | カスタマイズ |
| --- | --- |
| 構築中（全試合が未開始） | 自由に変更可能 |
| 準備完了（全SEEDスロット充足、全試合未開始） | 変更可能（チーム配置をリセット） |
| 進行中（試合にスコアが1つでも入っている） | **変更不可** |
| 完了（全試合 FINISHED） | **変更不可** |

判定は「所属 matches にスコアが入った match_entries が1つでもあるか」で行う。シンプルなクエリ1つで判定可能。

**ブラケット間の制限:**

あるSUBブラケットが `source_match_id` でMAINブラケットの試合を参照している場合、
MAINブラケットの該当試合を削除するにはSUBブラケット側の参照も解除する必要がある。

### 8-4. バリデーション

- 各試合に `tournament_slots` が正確に2つ存在すること
- `source_match_id` が同一competition内の試合を参照していること
- DAG（有向非巡回グラフ）であること（ブラケット間の参照を含む循環参照がないこと）
- `seed_number` がブラケット内でユニークであること
- 各ブラケット内に最終試合（MATCH_WINNERで後続されない試合）が正確に1つ存在すること

# トーナメント機能 実装設計方針書（3/3: 運用・テスト編）

## 9. スコア修正制限

### 9-1. ブラケット内のスコア修正

| 状態 | 修正可否 |
| --- | --- |
| 修正対象の試合より後の試合（同一ブラケット内）がすべて未開始 | 修正可能 |
| 後続試合が1つでも開始済み | 修正不可 |

### 9-2. ブラケット間のスコア修正制限

MAINブラケットの試合を修正する場合、その試合を `source_match_id` で参照している
**SUBブラケットの試合も考慮する**。

例: 準決勝1のスコアを修正したい場合、3位決定戦ブラケットの試合が開始済みなら修正不可。

### 9-3. 修正可否の判定アルゴリズム（グラフ探索）

スコア修正の可否判定は、修正対象の試合から**ブラケットをまたいでグラフを再帰的に探索**する必要がある。

```
function canModifyScore(matchId):
  // この試合を source_match_id で参照している全スロットを取得
  // （MATCH_WINNER / MATCH_LOSER 両方、ブラケット間含む）
  dependentSlots = SELECT * FROM tournament_slots
                   WHERE source_match_id = matchId

  for each slot in dependentSlots:
    dependentMatch = slot に紐づく match_entry の試合
    if dependentMatch のスコアが入っている:
      return false  // 修正不可
    // さらにその試合の先も再帰的にチェック
    if not canModifyScore(dependentMatch.id):
      return false

  return true  // 全ての後続試合が未開始なら修正可能
```

**修正が可能な場合の巻き戻し処理:**

```
function rollbackFromMatch(matchId):
  // 1. この試合の winner_team_id をクリア
  // 2. この試合を参照する全スロットの match_entry.team_id をクリア
  // 3. さらにその先も再帰的にクリア（ブラケット間含む）
  dependentSlots = SELECT * FROM tournament_slots
                   WHERE source_match_id = matchId
  for each slot in dependentSlots:
    slot に紐づく match_entry の team_id を NULL に戻す
    dependentMatch = その match_entry の試合
    rollbackFromMatch(dependentMatch.id)  // 再帰
```

この再帰探索により、準決勝1のスコアを修正した場合、「決勝のスロット1のチーム」と「3位決定戦のスロット1のチーム」が同時にクリアされる。

### 9-4. 進出先との連動

リーグ要件書（セクション4-6）と同じ:

| 進出先の状態 | トーナメントのスコア修正 |
| --- | --- |
| 進出先の試合が1つも動いていない | 修正可能（順位再導出 → エントリー再構成） |
| 進出先の試合が1つでも動いている | 修正不可 |

**トーナメントが複数の進出先を持つ場合**（例：1位→決勝リーグX、3位→順位決定リーグY）、**いずれか1つでも進出先の試合が動いていれば修正不可**とする。

## 10. GraphQL API 設計方針

### 10-1. Query

- **competition単位のブラケット全体像**: 全ブラケット + 試合 + スロット情報を一括取得
- **ブラケット単位の試合一覧**: 特定ブラケットの試合とスロット情報
- **competition単位の順位**: 全ブラケットの結果を統合して算出
- **ブラケットの状態**: 各ブラケットの状態（試合・スロットから導出）と進行率

### 10-2. Mutation

- **ブラケット自動生成**: チーム数 + オプション（3位決定戦等）でMAIN + SUBを生成
- **ブラケットの追加・削除**: SUBブラケットの手動追加・削除
- **ブラケットカスタマイズ**: 試合追加/削除、スロット接続変更
- **スコア入力**: 試合のスコアを入力
- **スコア修正**: セクション9の制限付き

### 10-3. 副作用の連鎖

リーグ要件書セクション8-3と対応する形で整理する。

| トリガー | 連鎖する処理 |
| --- | --- |
| 試合スコアの確定 | winner導出 → 後続スロットへのチーム配置（**ブラケット間含む**） → 後続試合が開始可能に → **全試合完了チェック（下記へ連鎖）** |
| 全試合完了チェックでcompetition完了を検出 | 順位確定 → `promotion_rules` に基づきエントリー作成（進出先へ）→（期待チーム数到達なら）スロット割当（プリセット時）→ 進出先がリーグなら試合生成、進出先がトーナメントならSEED配置完了で開始可能に |
| トーナメントのスコア修正（進出先が未稼働の場合のみ） | 巻き戻し（グラフ再帰探索、セクション9-3参照）→ 順位再導出 → エントリー再構成 |
| 進出ルールの変更（進出先が未稼働の場合のみ） | 期待チーム数の再算出 → SEEDスロットの team_id クリア → 再配置 |
| ブラケットカスタマイズ | 整合性バリデーション → 期待チーム数の再算出 |

## 11. 実施要項の変則パターンへの対応例

### バスケットボール（晴天時）: 4チーム + 3位決定戦

自動生成で対応可能。MAINブラケット + SUBブラケット（3位決定戦）。

```
competition: バスケ決勝
├── tournament MAIN: seed1×seed4, seed2×seed3, 勝者同士で決勝
└── tournament SUB: 準決勝敗者同士で3位決定戦
```

### ビーチボール（晴天時）: 12チーム変則ブラケット

手動カスタマイズが必要。MAINブラケットを12チーム対応で構築。

### ドッジビー: 3リーグ + 敗者復活枠

MAINブラケット + SUBブラケット（敗者復活トーナメント）として構成。
SUBブラケットの `MATCH_LOSER` スロットがMAINブラケットの特定ラウンドの敗者を受け取る。

### 8チーム大会で5-8位も決めたい場合

```
competition: サッカー決勝
├── tournament MAIN: 8チームシングルエリミ（準々→準決→決勝）
├── tournament SUB "3位決定戦": 準決勝の敗者2チーム
└── tournament SUB "5-8位決定戦": 準々決勝の敗者4チームでミニトーナメント
```

## 12. テスト観点

### ブラケット構造

- 4チーム、8チームブラケットが正しく自動生成される
- `tournament_slots` の source_type, source_match_id が正しく接続される
- seed_number がブラケット内でユニーク
- DAGの循環参照がバリデーションで検出される（ブラケット間含む）
- 各ブラケット内の最終試合（MATCH_WINNERで後続されない試合）がグラフ構造から正しく導出される

### マルチブラケット

- MAINブラケットとSUBブラケットが正しく生成される
- 3位決定戦のスロットがMAINブラケットの準決勝を `source_match_id` で参照している
- 5-8位決定戦のスロットが準々決勝を参照している
- SUBブラケットの追加・削除ができる
- MAINブラケットの試合削除時にSUB側の参照がバリデーションで検出される

### ブラケット間自動進行

- MAINブラケットの準決勝完了時に、決勝に勝者が配置される（同一ブラケット内）
- **同時に、3位決定戦ブラケットに敗者が配置される**（ブラケット間）
- 3位決定戦ブラケットの両スロットが埋まった時点で試合が開始可能になる
- ブラケット間進行に追加のロジックが不要なこと（同一ブラケット内と同じコードパスで動く）

### BYE処理

- 5チーム: BYE 3個。1回戦は1試合のみ。seed 1, 2, 3 は2回戦のSEEDスロットに直接配置
- 6チーム: BYE 2個。1回戦は2試合。seed 1, 2 は2回戦に直接配置
- 7チーム: BYE 1個。seed 1 は2回戦に直接配置
- 3チーム: BYE 1個。seed 1 は決勝に直接配置
- BYE試合が作成されないこと
- BYEのあるブラケットで期待チーム数が正しく算出される

### ブラケット状態の導出

- SEEDスロット未充足のブラケットが「構築中」と判定される
- 全SEEDスロット充足 + 全試合未開始のブラケットが「準備完了」と判定される
- 1つ以上の試合が開始済みのブラケットが「進行中」と判定される
- 全試合 FINISHED のブラケットが「完了」と判定される
- SUBブラケット: MATCH_LOSER でチームが配置されるまで試合開始不可
- competition全体: 全ブラケットの全試合が FINISHED で完了と判定される

### 順位確定

- MAINブラケットのみ: 決勝勝者=1位、敗者=2位、準決勝敗者=同率3位
- 3位決定戦あり: 3位決定戦勝者=3位、敗者=4位
- 5-8位決定戦あり: 各ブラケットの結果が正しく統合される
- competition全体の順位が全ブラケットの結果から導出される
- SUBブラケットなしで同率3位のブラケットに `rank_spec = "3"` の進出ルールを設定しようとするとバリデーションエラー
- SUBブラケット（3位決定戦）追加後は `rank_spec = "3"` が設定可能

### 進出システムとの接続

- MAINブラケットのSEEDスロットに進出ルール経由でチームが配置される
- 全SEEDスロット充足でMAINブラケットが開始可能になる
- **スコア入力ごとに全試合完了チェックが走り**、全完了で進出処理が開始されること
- 全試合完了前のスコア入力では進出処理がトリガーされないこと
- 進出ルールの rank_spec から算出されるチーム数 + 手動エントリー数 と SEEDスロット数が不一致の場合にバリデーションエラー
- `rank_spec = "1-2"` でMAIN決勝完了時に、3位決定戦（SUB）が未完了でも進出トリガーが発動すること
- `rank_spec = "1-3"` で3位決定戦（SUB）が未完了の場合は進出トリガーが発動しないこと

### 全4パターンの進出

- リーグ → リーグ: エントリー作成 + 試合自動生成が動作すること
- リーグ → トーナメント: SEEDスロットへの team_id 配置が動作すること
- トーナメント → トーナメント: ブラケット結果から順位導出 → SEEDスロット配置が動作すること
- トーナメント → リーグ: ブラケット結果から順位導出 → エントリー作成 + 試合生成が動作すること
- 進出先がトーナメントの場合、placement_method に応じた seed_number 割当が正しく動作すること
- 進出先がリーグの場合、tournaments テーブルが存在せずスロット配置なしでエントリーのみであること
- トーナメントが進出元の場合、promotion_rules.source_competition_id に competition_id を設定できること

### 配置制御

- MANUAL: promotion_rules.slot で指定した seed_number にチームが配置される
- MANUAL: rank_spec は単体指定のみ許可される（1レコード = 1順位 × 1スロット）
- SEED_OPTIMIZED: 全チーム揃った時点で上位シード同士がブラケット終盤まで当たらない位置に配置される
- BALANCED: 全チーム揃った時点で上位と下位が序盤で対戦する形に配置される
- RANDOM: 全チーム揃った時点でseed_number がランダムに割り当てられる
- プリセットのスロット割当は期待チーム数到達時に実行される（エントリー作成時ではない）
- placement_method は tournaments テーブル（MAINブラケット）に設定され、SUBブラケットではNULL
- リーグ進出時はスロット配置なしでエントリーのみ作成される

### 進出ルール変更制限

- 進出先トーナメントの全試合が未開始の場合、進出ルールを変更できる
- 進出先トーナメントにスコアが1つでも入っている場合、進出ルールを変更できない
- ルール変更時にSEEDスロットの team_id がクリアされ、新ルールで再配置される
- ルール変更時にブラケット構造（matches + tournament_slots）自体は変更されない

### 手動エントリー

- 全試合未開始の場合、管理者がSEEDスロットに直接チームを配置できる
- 試合にスコアが入っている場合、手動エントリーの追加・削除が不可になること

### 子リーグのスコア修正制限

- 進出先トーナメントの全試合が未開始の場合、子リーグのスコアを修正できる
- スコア修正後、進出チームが再計算されSEEDスロットが再配置される
- 進出先トーナメントにスコアが1つでも入っている場合、子リーグのスコアを修正できない

### スコア修正

- 同一ブラケット内: 後続試合未開始なら修正可能
- **ブラケット間**: SUBブラケットの試合が開始済みならMAINブラケットの修正不可
- 進出先の試合が動いていれば修正不可
- **複数の進出先がある場合**: いずれか1つでも進出先の試合が動いていれば修正不可
- 修正可否判定がブラケットをまたいで再帰的にグラフ探索されること
- 修正時の巻き戻しが決勝とSUBブラケットの両方のスロットをクリアすること

### 引き分け禁止

- トーナメントの試合で両チーム同点のままFINISHEDにできないこと
- winner_team_id が NULL のままFINISHEDにできないこと
- スコア同点でも管理者が手動でwinner_team_idを指定できること（PK戦等）

### 不戦勝

- 棄権チーム指定 → winner_team_id 設定 → FINISHED → 自動進行がトリガーされること
- 不戦勝でも後続の MATCH_WINNER / MATCH_LOSER スロットに正しくチームが配置されること
- 不戦勝の処理に特別な分岐が不要なこと（通常の自動進行と同じコードパス）

### 同時実行制御

- 2つのスコア入力がほぼ同時に行われても、スロットの team_id が二重書き込みされないこと
- 自動進行処理がトランザクション内で実行され、途中失敗時にロールバックされること

### 試合削除制約

- 他のスロットから source_match_id で参照されている試合を削除しようとするとエラー（RESTRICT）
- 参照元スロットの接続を解除してから試合を削除できること

### 既存機能への非影響

- リーグの試合が従来通り動作する（matches テーブルに変更なし）
- match_entries の既存動作に影響がない

### データ整合性バリデーション

- `source_match_id` に別competitionの試合IDを指定するとバリデーションエラーになること
- トーナメント試合作成時に match + 2 match_entries + 2 tournament_slots が必ずセットで作成されること
- 1つの試合に tournament_slots が1つしかない状態がバリデーションで検出されること
- ブラケット削除時に tournament_slots が CASCADE で削除されること
- ブラケット削除時にアプリ層で所属 matches も正しく削除されること（セクション3-4の手順）
- `source_match_id` で参照されている試合の削除が RESTRICT で拒否されること
- 進出ルールが competition_id + bracket_type=MAIN で正しくMAINブラケットのSEEDスロットを特定すること
- 同一competition内にMAINブラケットを2つ作成しようとするとバリデーションエラーになること

## 用語整理

| 用語 | 意味 |
| --- | --- |
| competition | UIレベルの「1つのトーナメント」。`competitions` テーブルの `type = 'TOURNAMENT'` |
| ブラケット（tournament） | 個別のブラケットツリー。1つのcompetition内に複数存在できる |
| MAINブラケット | メインブラケット。外部からの進出（SEEDスロット）を受け入れる。`bracket_type = 'MAIN'` |
| SUBブラケット | サブブラケット。3位決定戦、5-8位決定戦、敗者復活戦、ダブルエリミの敗者ブラケット等すべて。`bracket_type = 'SUB'`。MAINとの違いはラベルのみで、データ構造は同一 |
| ブラケットの状態 | `tournaments` テーブルにはステータスカラムを持たない。所属する試合とスロットの状態から導出する（構築中/準備完了/進行中/完了） |
| 最終試合 | ブラケット内で、自分を `source_match_id` として参照する `MATCH_WINNER` スロットが存在しない試合。DBにフラグは持たず、グラフ構造から導出する |
| スロット（tournament_slot） | 試合の各側のソース情報。match_entries を1:1拡張するテーブル |
| source_type | スロットのソース種別。SEED / MATCH_WINNER / MATCH_LOSER |
| SEED | 外部からの配置。進出システムの「スロットN番に配置」に対応 |
| MATCH_WINNER | 前の試合の勝者が、このスロットの `match_entry.team_id` に書き込まれる。通常は同一ブラケット内で使用するが、ダブルエリミネーション等ではブラケット間でも使用可能 |
| MATCH_LOSER | 前の試合の敗者が、このスロットの `match_entry.team_id` に書き込まれる。**ブラケット間接続の主な手段** |
| seed_number | SEEDスロットのシード番号。ブラケット内でユニーク |
| ブラケット間接続 | `source_match_id` で別ブラケットの試合を参照すること |
| DAG | 有向非巡回グラフ。循環参照がないグラフ構造 |
| BYE | 初戦免除。BYE試合は作成せず、BYE対象チームを次ラウンドのSEEDスロットに直接配置する |
