# リーグ機能 実装要件書

## 本ドキュメントの目的

リーグ機能のリファクタリング・再実装にあたり、実装者が判断に迷わないよう、確定済みの要件・ルール・前提条件をまとめたもの。
技術的な設計判断（DB スキーマの具体構造、関数の分割粒度など）は実装者に委ねる。既存コードのうち良い実装はそのまま利用し、構造が要件と合わない部分やコードが汚い部分は変更を加える方針。

---

## 1. 順位決定ルール

### 1-1. 基本方針

- 順位決定に使う条件（勝ち点、得失点差、総得点など）は**競技ごとに異なる**。
- 条件は**優先順位付きのリスト**として管理する。1番目の条件で並べ、同値なら2番目の条件で比較、という形で順位を確定していく。
- 管理画面上で、条件の**優先順位を並べ替えられる**ようにする（ドラッグ&ドロップ等）。
- 順位決定ルールは**競技（sports）単位**で設定する。同じ競技内のリーグ（予選A、予選B、決勝など）はすべて同じルールが適用される。

### 1-2. 順位決定条件の候補（マスタ）

システムが持つ順位決定条件の候補は以下の **5つ** に固定する:

| 条件キー | 意味 | 算出方法 |
| --- | --- | --- |
| **勝ち点** | 勝ち/引き分け/負けに応じたポイント合計（チーム間で消化試合数に差がある場合は平均） | 試合結果 × ポイント設定から自動算出 |
| **得失点差** | 総得点 − 総失点（チーム間で消化試合数に差がある場合は平均） | 試合結果から自動算出 |
| **総得点** | 全試合の得点合計（チーム間で消化試合数に差がある場合は平均） | 試合結果から自動算出 |
| **当該対戦結果** | 同順位チーム同士の直接対決の勝敗 | 該当試合を参照して判定 |
| **管理者裁定** | じゃんけん等の外部手段を経て、管理者が最終順位を決定する | **管理者が結果を手動入力** |

### 1-3. 競技ごとの条件順序の設定

上記の候補から、競技ごとに**使用する条件とその優先順位を順序つきで保存**する。

設定例:

| 競技 | 設定する順序 |
| --- | --- |
| サッカー | 勝ち点 → 得失点差 → 総得点 |
| バスケ | 勝ち点 → 総得点 |
| バレー | 勝ち点 → 当該対戦結果 → 得失点差 |
| ドッジビー | 勝ち点 → 得失点差 → 総得点 → 管理者裁定 |
- 候補5つのうち、**全部を使う必要はない**。競技ごとに必要な条件だけを選んで並べる。
- **最低1つは設定必須**とする（バリデーション）。
- 順序は管理画面上で並べ替えられる。
- **勝ち点を1番目にする制約はない。** 自由に並べ替え可能（例：「総得点 → 勝ち点」も設定可能）。

### 1-4. 条件ごとの実装上の注意

**当該対戦結果:**

- 同順位が2チームの場合: その2チームの直接対決の勝者を上位とする。直接対決が引き分けだった場合は分離できないため、次の条件へ進む。
- 同順位が3チーム以上の場合: 当該チーム間の直接対決の試合だけを取り出し、その結果から勝ち点を再集計して順位を決める。例えばA・B・Cが同順位なら、A-B、B-C、A-Cの3試合の結果だけで勝ち点を再計算する。
- 再集計で一部だけ分離できた場合（例：Aだけ抜けてB・Cがまだ同順位）、分離できなかったチーム群（B・C）だけを対象に次の条件へ進む。
- 再集計しても全く分離できなかった場合も、その同順位チーム群のまま次の条件へ進む。

**管理者裁定:**

- システム上は自動判定できないため、管理者が結果を入力する形とする。
- 入力方法は**同順位チーム間のタイブレーク優先度**。管理者はタイになっているチーム同士のどちらが上かを決めるだけで、何位かはシステムが順位計算で導出する。
- **裁定を通常の試合として作成してはならない。** 勝ち点が加算されて確定済みの他チームの順位が崩れるため。裁定結果は通常の順位計算とは別枠で保存する。
- **「管理者裁定待ち」の状態はDBに永続化しない。** 順位計算ロジックが全条件を適用しても同順位が残っていて、かつタイブレーク優先度が保存されていなければ「裁定待ち」と判断できる。フロント側はこの順位計算結果を見て裁定入力UIを出すかどうかを判断する。
- **同一リーグ内で複数のタイグループが発生する場合がある**（例：2-3位がタイ、かつ5-6位もタイ）。各グループは独立して裁定結果を入力できるようにする。

### 1-5. 最終的な同順位の解消（管理者裁定）

**設定された順位決定条件に管理者裁定が含まれていてもいなくても、すべての条件を適用して同順位が残った場合は、最終手段として必ず管理者裁定で決着する。**

- 管理者裁定は必ず決着がつくため、「タイのまま確定」という状態は発生しない。
- 条件リストに管理者裁定を明示的に含めた場合は、その優先順位の位置で発動する。含めなかった場合でも、暗黙の最終条件として機能する。
- 自動進出は、**進出対象の順位が確定していれば実行される**。進出対象外の順位に裁定待ちが残っていても進出を妨げない。

## 2. 勝ち点設定

### 2-1. ポイント設定

- 勝ち・引き分け・負けそれぞれに付与するポイントを**リーグ（DB）に紐づけて**設定できるようにする。
- デフォルト値の例: 勝ち = 3、引き分け = 1、負け = 0

### 2-2. 消化試合数に差がある場合の平均化

**チーム間で消化試合数に差がある場合**、以下の指標を平均化して順位判定に使用する。奇数リーグに限らず、偶数リーグでも進行状況によって消化試合数に差が出る場合は同様に平均化する。

| 指標 | 平均化の方法 |
| --- | --- |
| 勝ち点 | 総勝ち点 ÷ 消化試合数 |
| 得失点差 | 総得失点差 ÷ 消化試合数 |
| 総得点 | 総得点 ÷ 消化試合数 |
- **BYE（対戦相手がいない回）は消化試合数に含めない。**
- BYE 以外の試合（通常の試合・不戦勝・不戦敗すべて）は消化試合数に含める。
- 全チームの消化試合数が同じになった場合（全日程終了後）は、平均ではなく通常の合計値で順位を判定する。

## 3. 順位表の算出

### 3-1. 基本方針

- 順位表は**試合結果とリーグ設定から都度導出**する。順位表をDBに永続化せず、Queryが呼ばれるたびに試合結果から計算して返す。1リーグ最大8チーム程度・同時アクセス数十人規模のため、都度計算でパフォーマンス上の問題はない。
- **途中経過でも暫定順位を表示する。** 全試合が終わっていない状態でも、その時点の試合結果から順位を計算して返す。
- 暫定順位では同順位（タイ）が発生してよい。同順位の場合、次の順位はその分飛ばす（例：4位、4位、6位）。
- **管理者裁定が必要かどうかは順位計算結果から判断する。** 全試合終了後に全条件を適用しても同順位が残り、かつタイブレーク優先度が保存されていなければ「裁定待ち」。専用のステータスフラグはDBに持たない。
- **既存の `league_standing` 関連の実装（テーブル・ロジック・`calculateLeagueStandings` 含む）はすべて削除する。** テーブルの削除はマイグレーションコマンドを使って行うこと（直接ファイルをいじって変更しない）。
- 順位計算は新規に実装し直す。上記の順位決定ルール・勝ち点設定・消化試合数差の平均化対応を反映した形にする。

### 3-2. 算出に必要な入力

- 全試合結果（スコア、勝敗）
- リーグのポイント設定（勝ち/引き分け/負けの各ポイント）
- リーグの順位決定ルール（競技に紐づく条件リストと優先順位）
- リーグのチーム数と各チームの消化試合数（平均化が必要かの判定用）
- 管理者裁定のタイブレーク優先度（保存済みの場合）

### 3-3. 算出の流れ

1. 各チームの成績を集計する（勝ち点、得点、失点、得失点差、試合数など）
2. チーム間で消化試合数に差がある場合、勝ち点・得失点差・総得点を平均に変換する
3. 順位決定ルールの優先順位に従ってソートする
4. 同順位がある場合、次の順位はその分飛ばす（例：4位、4位、6位）
5. **全試合が完了状態の場合のみ**: 同順位が残っていれば、保存済みのタイブレーク優先度を適用して解消する。優先度が未保存であれば同順位のまま返す（フロント側が裁定入力UIを表示する判断材料になる）

### 3-4. 自動進出のトリガー判定

- 自動進出の起点は「最後の試合スコア入力」ではなく、**リーグ内の全試合のステータスが「完了」になっているかどうか**で判定する。
- スコアは順番通りに入力されるとは限らないため、毎回のスコア入力時に全試合の状態をチェックし、全完了であれば進出処理を開始する。

## 4. 自動進出

### 4-1. 基本方針

- 予選リーグの**全試合スコアが揃い、進出対象の順位が確定した時点で自動実行**する。管理者裁定待ちがある場合はその結果入力後に実行される。管理者の確認ステップは挟まない。
- 進出先への**エントリー作成と試合（対戦カード）作成の両方**を自動で行う。

### 4-2. 対応する進出パターン

| 進出元 | 進出先 | 今回のスコープ |
| --- | --- | --- |
| リーグ | リーグ | **実装する** |
| リーグ | トーナメント | **実装する** |
| トーナメント | トーナメント | 将来対応（拡張可能な設計にしておく） |
| トーナメント | リーグ | 将来対応（拡張可能な設計にしておく） |

進出の仕組み自体は進出元・進出先の種類に依存しない汎用的な設計にし、将来的に全4パターンが同じロジックで動くようにする。

### 4-3. 進出ルールの設定

- 進出ルールは**子リーグ側**で設定する。子リーグが「自分のリーグの何位を、どの進出先のどのスロットに送るか」を定義する形。
- 親リーグ（進出先）側は設定を持たない。親リーグは `promotion_rules` で自分が `target_competition_id` として登録されているレコードを集計して期待チーム数を算出する（4-4参照）。
- 各ルールには以下を設定できるようにする:
    - **進出元リーグ**: このルールを持つリーグ自身（暗黙的に決まる）
    - **対象順位**: 何位のチームが進出するか（下記の3パターンで指定可能）
    - **進出先**: 進出先のリーグまたはトーナメント
    - **進出先でのスロット位置**（任意）: 進出先のどの位置に配置するか。進出先の配置方法が「手動配置」の場合にのみ使用する。手動配置では `rank_spec` を単体指定にし、1レコードにつき1スロットを指定する形とする。リーグ進出時は不要（単にエントリーされる）。

**対象順位の指定方法:**

| 指定方法 | 例 | 意味 |
| --- | --- | --- |
| 単体指定 | `1` | 1位のみ進出 |
| 複数指定 | `1, 3` | 1位と3位が進出 |
| 幅指定 | `1〜4` | 1位から4位まで進出 |
- いずれの指定方法でもリーグ進出・トーナメント進出の両方で使用可能。
- **ただし、進出先の配置方法が「手動配置」の場合**、`rank_spec` は単体指定のみ使用可能（1レコードにつき1順位×1スロット）。幅指定・複数指定を使いたい場合はプリセット配置を選択すること。

設定例:

```
【決勝トーナメント（配置方法: 手動配置）の場合】
リーグAの1位     → 決勝トーナメント スロット1
リーグAの2位     → 決勝トーナメント スロット4
リーグBの1位     → 決勝トーナメント スロット2
リーグBの2位     → 決勝トーナメント スロット3

【決勝トーナメントY（配置方法: シード最適化）の場合】
リーグDの1〜4位  → 決勝トーナメントY（スロットは自動割当）

【決勝リーグX（リーグなので配置方法なし）の場合】
リーグCの1〜3位  → 決勝リーグX（単にエントリー）
```

### 4-4. 自動進出の仕組み（2段階）

進出処理は**エントリー作成**と**試合生成**の2段階に分離する。

**ステップ1: エントリー作成（子リーグ終了時）**

- 子リーグの全試合スコアが揃い、**進出対象の順位が確定した**時点で実行する（管理者裁定待ちがあればその結果入力後）。進出対象外の順位に管理者裁定待ちが残っていても、対象順位が確定していれば進出処理を実行する。
- 対象順位のチームを特定し、進出先にエントリーを作成する。
- 進出先の配置方法が「手動配置」の場合、`promotion_rules` で指定されたスロットにチームを配置する。
- 進出先の配置方法がプリセットの場合、この時点ではスロット割当は行わない（全チームが揃ってから実行する）。
- この時点では試合は作成しない。

**ステップ2: 試合生成（期待チーム数到達時）**

- 進出先（親リーグ/トーナメント）は、`promotion_rules` テーブルで自分の `competition_id` が `target_competition_id` として登録されているレコードを参照し、各レコードの `rank_spec` から何チーム来るかを合算して**期待チーム数**を算出する。
- 例: リーグAが上位2チーム、リーグBが上位2チームを親リーグXに送る設定になっている → 親リーグXの期待チーム数は4。
- **手動で直接エントリーされたチームも期待チーム数に含める。** 運営が後から手動でチームを追加する場合もあるため、期待チーム数は「進出ルールによる自動エントリー数 + 手動エントリー数」で算出する。
- エントリー数が期待チーム数に達した時点で、以下を実行する:
    - 進出先の配置方法がプリセットの場合、全チームに対してプリセットのロジックでスロットを割り当てる。
    - 対戦カード（試合）を自動生成する。

**この設計のメリット:**

- リーグ・トーナメント問わず同じロジックで統一できる。
- 後から進出ルールを追加した場合、期待チーム数が自動で変わるため、特別な再計算が不要。
- 各リーグが独立して終了でき、全リーグの終了を待つ必要がない。

### 4-5. 進出ルールの変更制限

進出ルールの追加・変更・削除は、**進出先の状態**によって制限する。

| 進出先の状態 | ルール変更 |
| --- | --- |
| エントリー数が期待チーム数に達していない（試合未生成） | **変更可能** |
| 試合が生成済みだがスコアが1つも入力されていない | **変更可能**（試合を削除して再生成する） |
| 試合のスコアが1つでも入力されている | **変更不可** |
- 変更可能な状態でルールが変更された場合、期待チーム数が再算出され、必要に応じてエントリーや試合が再構成される。
- 一度でもスコアが入力された進出先は、進出ルールがロックされる。
- **手動エントリーの追加・削除も同じ制限に従う。** 試合のスコアが入っている進出先には手動エントリーを追加できない。

### 4-6. 子リーグのスコア修正制限

子リーグ（進出元）の試合スコア修正も、**進出先の状態**によって制限する。

| 進出先の状態 | 子リーグのスコア修正 |
| --- | --- |
| 進出先の試合が1つも動いていない（スコア未入力） | **修正可能**（順位再計算 → エントリー再構成） |
| 進出先の試合が1つでも動いている | **修正不可** |
- スコア修正が可能な場合、修正後に順位が再計算され、進出エントリーや試合も再構成される。
- **子リーグが複数の親リーグに進出している場合**（例：1位→決勝リーグX、3位→順位決定リーグY）、いずれか1つでも進出先の試合が動いていれば修正不可とする。
- 運用上、予選リーグの結果を確定させてから決勝に進む手順を踏むため、この制限で問題は防げる。

## 5. 配置制御（スロット設定）

### 5-1. 基本方針

- 自動進出時、進出先での組み合わせ位置を制御できるようにする。
- 配置方法は**進出先（トーナメント）側の設定**として持つ。同じトーナメントに進出する全チームに対して1つの配置方法が適用される。
- **プリセット（定義済みの配置パターン）から選択する方法**と、**手動で直接スロットを指定する方法**の両方に対応する。

### 5-2. 配置プリセット

以下の3種類をプリセットとして用意する。進出先のトーナメント設定時に選択できるようにする。

**シード最適化:**
上位シードのチーム同士がトーナメント終盤まで当たらないように配置する。1位と2位は反対ブロックに入り、決勝まで対戦しない形になる。

**均等配置:**
シード順で対称的に組み合わせる。1vs8、2vs7、3vs6、4vs5のように、上位と下位が序盤で対戦する形になる。

**ランダム配置:**
スロットをランダムに割り当てる。シードや順位を考慮しない。

### 5-3. 手動配置

プリセットで対応できない場合は、進出先の配置方法を「手動」に設定する。この場合、各 `promotion_rules` レコードで個別にスロット位置を直接指定する（`rank_spec` は単体指定のみ）。

### 5-4. 設計上の注意

- 配置方法は**進出先ごとに1つ**。同じトーナメントに進出する全チームに同じ配置方法が適用される。
- スロットは**トーナメント進出時に使用する**。リーグ進出時はスロット不要で、単にエントリーするだけ。
- プリセットの配置ロジックはバックエンド側で実装する。フロントはプリセット名を選択するだけでよい。
- プリセットの場合、全チームが揃った時点（期待チーム数到達時）でスロット割当を実行する。
- **複数リーグから進出する場合のシード順:** 自動生成時はエントリーの作成順（ULID順）でシード番号を1から振る。管理者が手動で位置を変更した場合は、左から順に1, 2, 3... と番号を振り直す。
- **将来的にリーグ→リーグ、リーグ→トーナメント、トーナメント→トーナメント、トーナメント→リーグの全パターンで自動進出が使えるようにする。** 今回はリーグ→リーグ、リーグ→トーナメントを実装するが、進出の仕組み自体は進出元・進出先の種類に依存しない汎用的な設計にしておくこと。

## 6. テーブル設計方針

### 6-1. 新規テーブル

今回新たに必要なテーブルは以下の3つ。

**ranking_rules（順位決定ルール）**

競技ごとの順位決定条件と優先順位を保存する。`sport_id` と `condition_key` の複合主キーとする（他テーブルからの参照がなく、設定変更時は全削除→全挿入で使うため、単独の `id` は不要）。

| カラム | 内容 |
| --- | --- |
| sport_id（PK） | 競技ID（FK → sports） |
| condition_key（PK） | 条件キー（WIN_POINTS / GOAL_DIFF / TOTAL_GOALS / HEAD_TO_HEAD / ADMIN_DECISION） |
| priority | 優先順位（1, 2, 3...） |
- `(sport_id, priority)` にユニーク制約を付けて同一競技内の優先順位重複を防ぐ。
- 現在の実装が `league_id` で紐づいている場合は `sport_id` に変更すること。

**tiebreak_priorities（タイブレーク優先度）**

管理者裁定の結果を保存する。同順位チーム間の上下関係のみを記録する。`league_id` と `team_id` の複合主キーとする（他テーブルからの参照がなく、リーグ×チームで一意に特定できるため、単独の `id` は不要）。

| カラム | 内容 |
| --- | --- |
| league_id（PK） | リーグID（FK → leagues） |
| team_id（PK） | チームID（FK → teams） |
| priority | 同順位チーム内での優先度（数値が小さいほど上位） |
- 順位そのものは保存しない。「同順位チーム間でどちらが上か」だけを記録する。
- 裁定待ちのステータス（PENDING / RESOLVED）は持たない。タイブレーク優先度が保存されているかどうかで判断する。
- 裁定を通常の試合として作成してはならない（勝ち点が加算されて他チームの順位が崩れる）。

**promotion_rules（進出ルール）**

子リーグ側に設定する進出ルール。FK先が `competitions` なので全進出パターンに対応できる。

| カラム | 内容 |
| --- | --- |
| id | 主キー |
| source_competition_id | 進出元の大会ID（FK → competitions） |
| target_competition_id | 進出先の大会ID（FK → competitions） |
| rank_spec | 進出対象順位（"1" / "1,3" / "1-4" 等の文字列。アプリ側でパース。進出先が手動配置の場合は単体指定のみ） |
| slot | スロット位置（進出先の配置方法が MANUAL の場合のみ使用。それ以外はNULL） |

### 6-2. 既存テーブルへの変更

**既存テーブルへのカラム追加はなし。**

- `leagues` テーブルには勝ち/引き分け/負けのポイントカラムが既に存在する。
- `placement_method`（配置方法）は `tournaments`（ブラケット）テーブルに持たせる。リーグ要件書のスコープではなく、トーナメント方針書側で定義する。

### 6-3. 既存テーブルの削除

- `league_standing` テーブル、`calculateLeagueStandings` を含む関連コード・ロジックは**すべて削除する**。
- テーブルの削除は**マイグレーションコマンドを使って行う**こと。直接ファイルを編集して変更しない。

### 6-4. `calculationType` の扱い

- 既存の `calculationType` は、今回の順位決定ルール（競技単位の優先順位付きリスト）に置き換える方向で整理する。
- 既存データとの互換性が必要かどうかは、実装者が既存コードを確認した上で判断する。

### 6-5. テーブル設計の判断基準

今回の設計で適用した判断基準を以下にまとめる。今後テーブルを追加・変更する際も同じ基準で判断すること。

- **保存すべきもの:** 他から導出できない事実（試合スコア、管理者の設定、裁定結果など）
- **保存しなくていいもの:** 既存データから計算・カウントで出せるもの（チーム数、勝ち点合計、順位、裁定待ちステータスなど）
- **既存テーブルにカラム追加:** そのデータが親に対して1対1の関係（ポイント設定など）
- **新規テーブルを作る:** そのデータが親に対して1対多の関係（順位決定ルール、タイブレーク優先度など）
- **紐づけ先:** そのデータが変わる単位で決める（競技ごとに変わるなら `sport_id`、リーグごとに変わるなら `league_id`）
- **主キー:** 他テーブルからFKで参照されない設定系テーブルは複合主キーでよい。単独の `id` は不要。

## 7. 管理画面で必要な操作

以下の設定・操作を管理画面から行えるようにする:

1. **ポイント設定**: リーグごとの勝ち/引き分け/負けポイントの設定
2. **順位決定ルール**: 競技ごとに候補5条件からの選択・優先順位の並べ替え
3. **自動進出ルール**: 進出元リーグ・対象順位・進出先・スロット位置（手動配置時）の設定
4. **配置方法設定**: 進出先トーナメントごとにプリセット（シード最適化/均等配置/ランダム）または手動配置を選択
5. **タイブレーク優先度入力**: 同順位のチーム間の上下を決定する入力
6. **ステージ全体の可視化**: どのリーグからどこへ進出するかの全体像の確認

## 8. GraphQL API 設計方針

具体的なスキーマ設計は実装者に委ねるが、以下の方針は守ること。

### 8-1. Query（参照系）

- **順位表は Query で都度算出して返す。** DB に順位表を永続化しない。リクエストのたびに試合結果から計算する。
- **途中経過でも暫定順位を返す。** 同順位はそのまま返し、次の順位を飛ばす（例：4位、4位、6位）。
- 順位表のレスポンスには、各チームの成績（勝ち点、得点、失点、得失点差、消化試合数など）と順位を含める。
- 管理者裁定が必要かどうかは順位計算結果に同順位が残っているかどうかで判断できる。専用のフラグは不要。フロント側は全試合完了後の順位表に同順位が含まれていれば裁定入力UIを表示する。
- 進出先の期待チーム数と現在のエントリー数を参照できるようにする。

### 8-2. Mutation（更新系）

- **タイブレーク優先度の入力は Mutation で行う。** 入力後、順位確定 → エントリー作成 → （期待チーム数到達なら）試合生成までを一連の処理として実行する。フロントから複数回呼ぶ必要がない形にする。
- 進出ルールの追加・変更・削除も Mutation で行う。変更制限（4-5参照）のバリデーションはサーバー側で行う。
- **試合スコアの修正** は、進出先の試合が動いていない場合のみ許可する（4-6参照）。バリデーションはサーバー側で行う。
- ポイント設定、順位決定ルールの設定も Mutation で行う。

### 8-3. 副作用の連鎖

以下の処理は副作用として連鎖的に実行される。フロントから個別に呼ぶのではなく、トリガーとなる操作の中で自動的に走る形にする。

| トリガー | 連鎖する処理 |
| --- | --- |
| 試合スコアの入力（結果として全試合が完了状態になった場合） | 順位確定 → エントリー作成 → （期待チーム数到達なら）スロット割当（プリセット時）→ 試合生成 |
| 裁定結果（タイブレーク優先度）の入力 | 順位確定 → エントリー作成 → （期待チーム数到達なら）スロット割当（プリセット時）→ 試合生成 |
| 進出ルールの変更 | 期待チーム数の再算出 → 必要に応じてエントリー・試合の再構成 |
| 子リーグのスコア修正（進出先が未稼働の場合のみ） | 順位再計算 → エントリー再構成 → （期待チーム数到達なら）試合再生成 |

## 9. テスト観点

最低限、以下のケースで正しく動くことを確認する:

### 順位計算

- 勝ち / 引き分け / 負けの勝ち点が正しく計算される
- 消化試合数に差がある場合: 勝ち点・得失点差・総得点がすべて平均で順位判定される
- 全チーム同一消化試合数の場合: 合計値で順位が決まる（平均を使わない）
- 同勝ち点時: 順位決定ルールの優先順位に従って順位が決まる
- 全条件適用後の同順位: タイブレーク優先度が未入力なら同順位のまま返される
- 全条件適用後の同順位: タイブレーク優先度の入力後に順位が確定する
- 管理者裁定を条件に含めていないリーグでも、最終的に管理者裁定で解消される
- 暫定順位（途中経過）: 同順位が発生し、次の順位が飛ぶ（例：4位、4位、6位）

### 各順位決定条件

- **得失点差**: 同勝ち点のチームが得失点差で正しく分離される
- **総得点**: 同勝ち点・同得失点差のチームが総得点で正しく分離される
- **当該対戦結果（2チーム）**: 直接対決の勝者が上位になる
- **当該対戦結果（2チーム・引き分け）**: 分離できず次の条件へ進む
- **当該対戦結果（3チーム以上）**: 当該チーム間の直接対決の勝ち点再集計で順位が決まる
- **当該対戦結果（部分分離）**: 3チーム中1チームだけ分離でき、残り2チームが次の条件へ進む
- **管理者裁定**: 全条件適用後に同順位が残り、タイブレーク優先度の入力で順位が確定する
- **管理者裁定（複数グループ）**: 同一リーグ内で2箇所以上のタイが発生し、それぞれ独立して優先度を入力できる

### 競技ごとの差分

- 「勝ち点→得失点差→総得点」の競技で正しくソートされる
- 「勝ち点→総得点」のように条件を減らした競技で正しくソートされる
- 「勝ち点→当該対戦結果→得失点差」の順序で正しく判定される
- 同じ競技内の複数リーグ（予選A、予選B）で同じルールが適用される
- 順位決定ルールの優先順位を変更したら、順位が変わる
- 順位決定条件を0個で保存しようとするとバリデーションエラーになる

### 自動進出

- 全試合が完了状態になった時点でエントリーが作成される（試合はまだ作られない）
- 全試合完了前にスコア入力しても進出処理はトリガーされない
- 進出対象の順位に同順位が残っている場合はエントリーが作成されない
- 進出対象外の順位にのみ同順位が残っている場合、進出対象の順位が確定していれば進出が実行される
- タイブレーク優先度入力後にエントリーが作成される
- 期待チーム数が進出ルールから正しく算出される（手動エントリー分を含む）
- エントリー数が期待チーム数に達した時点で試合が自動生成される
- エントリー数が期待チーム数に達していない場合は試合が生成されない
- 対象順位の単体指定（1位のみ）で正しく進出する
- 対象順位の複数指定（1位, 3位）で正しく進出する
- 対象順位の幅指定（1〜4位）で正しく進出する
- リーグ → リーグ進出が正しく動作する（スロットなし、エントリーのみ）
- リーグ → トーナメント進出が正しく動作する（配置方法に従ってスロット配置）
- プリセット配置の場合、期待チーム数到達時にスロットが自動割当される
- 後から進出ルールを追加した場合、期待チーム数が更新される
- 進出先にスコアが入力されていない場合、進出ルールを変更できる
- 進出先にスコアが1つでも入力されている場合、進出ルールを変更できない

### スコア修正制限

- 進出先の試合が動いていない場合、子リーグのスコアを修正できる
- スコア修正後、順位が再計算されエントリーが再構成される
- 進出先の試合が1つでも動いている場合、子リーグのスコアを修正できない
- 複数の親リーグに進出している場合、いずれか1つでもロックなら修正できない
- 試合が動いている進出先に手動エントリーを追加できない

### 配置制御

- プリセット「シード最適化」: 全チーム揃った時点で上位シード同士が終盤まで当たらない位置に配置される
- プリセット「均等配置」: 全チーム揃った時点で1vs8、2vs7のように上位と下位が対称的に配置される
- プリセット「ランダム」: 全チーム揃った時点でスロットがランダムに割り当てられる
- 手動配置: 各 promotion_rules レコードで指定したスロット位置にチームが配置される
- 手動配置では rank_spec が単体指定のみ許可される
- 同じトーナメントに対する配置方法は1つだけ設定できる
- リーグ進出時は配置方法の指定が不要（単にエントリーされる）

### 副作用の連鎖

- 全試合完了 → 順位確定 → エントリー作成 → スロット割当（プリセット時）→ 試合生成が一連で実行される
- タイブレーク優先度入力 → 順位確定 → エントリー作成 → スロット割当（プリセット時）→ 試合生成が一連で実行される
- 進出ルール変更 → 期待チーム数再算出 → エントリー・試合の再構成が一連で実行される
- 子リーグスコア修正（進出先未稼働）→ 順位再計算 → エントリー再構成が一連で実行される

## 用語整理

| 用語 | 意味 |
| --- | --- |
| リーグ | 総当たり戦の予選グループ |
| 子リーグ | 進出元のリーグ。進出ルールを設定する側 |
| 親リーグ | 進出先のリーグ/トーナメント。子リーグからチームを受け入れる側 |
| トーナメント | 勝ち抜き戦（未実装、将来対応） |
| BYE | 奇数リーグで対戦相手がいない回。試合は発生しない。消化試合数に含めない |
| 不戦勝 | 相手がいるが出場しなかった場合の勝利。勝った側にDBで設定された勝ち点を付与する |
| 不戦敗 | 不戦勝の相手側。消化試合数にカウントする |
| 消化試合数 | 実際にプレーした（結果が記録された）試合の数。BYE は含まない |
| 勝ち点平均 | 総勝ち点 ÷ 消化試合数。チーム間で消化試合数に差がある場合に使用。得失点差・総得点も同様に平均化する |
| スロット | 進出先での配置位置（トーナメントのブラケット上の位置）。進出先の配置方法がプリセットなら全チーム揃った時点で自動割当、手動なら promotion_rules で個別指定 |
| 期待チーム数 | `promotion_rules` で自分が `target_competition_id` として登録されているレコードの `rank_spec` から算出される自動エントリー数 + 手動エントリー数。進出先に来る予定のチーム総数 |
| 順位決定ルール | 競技（sports）に紐づく優先順位付きの条件リスト。上から順に適用して順位を決める。同じ競技内のリーグはすべて同じルールが適用される |
| 順位決定条件の候補 | 勝ち点・得失点差・総得点・当該対戦結果・管理者裁定の5つ |
| 当該対戦結果 | 同順位チーム同士の直接対決の結果で順位を決める条件 |
| 管理者裁定 | 全条件適用後も同順位のチームに対し、管理者がタイブレーク優先度を入力して順位を決める条件。暗黙の最終条件としても機能する。裁定待ちの状態はDBに持たず順位計算結果から導出する |
